【目的】
- LLM比較のため、同一入力から「JUnitで実行可能なE2Eテストコード」を安定して生成する
- 仕様と実装の差分/不整合も検出する
- 無料モデルでもタイムアウトしにくいよう、簡潔に出力する

【重要方針】
- 長い説明文は避け、必要最小限で出力する
- 実装コードは、対象機能に関係する要素・バリデーション・遷移・エラーメッセージだけを優先して参照する
- 不明点は推測せず「不足情報」として短く列挙する
- Javaコードは「代表的な3〜4シナリオ」のみ実装する
- 1つのテストに複数観点を詰め込みすぎない

【指示】
1. 仕様書から対象機能の主要要件を把握する
2. 実装コードから対象機能に関係するUI要素、入力制約、状態遷移、エラーメッセージを抽出する
3. 仕様と実装の差分・不整合があれば列挙する
4. テスト設計は網羅性を意識しつつも簡潔にまとめる
5. 判断できない点は「不足情報」として示し、E2E設計への影響を一言で書く

【必須条件（テスト設計）】
- 合計8〜10件を目安に生成する
- 正常系 / 異常系 / 境界値 / セキュリティ を必ず含める
- 各カテゴリ最低1件以上
- 重複禁止
- 金融システム特有の観点（不正試行、金額境界、権限、二重送信、ロック、状態遷移）を優先する
- 各テストケースは2〜5行程度で簡潔に書く

【必須条件（JUnitテストコード）】
- JUnit 5（org.junit.jupiter）で実行できること
- Selenium 4（WebDriver）を前提にする
- ChromeDriver を使用（起動設定はコメントで可）
- 明示待ち（WebDriverWait）を使用し、固定sleep依存は避ける
- @BeforeEach / @AfterEach を用意し、driverの初期化/終了を行う
- 代表的な3〜4件だけ実装する
- 各テストは「1テスト=1シナリオ」を基本にする
- 期待結果は assert系で検証する
- ベースURLは System.getenv("BASE_URL") を優先し、なければ "http://localhost:8080" を使う
- Page Object は導入しなくてよい。1ファイルで簡潔にまとめる
- コードコメントは最小限にする

【出力形式】
(A) テスト設計（要点のみ）
テストケースID: TC-XXX
カテゴリ: [正常系/異常系/境界値/セキュリティ]
優先度: [高/中/低]
前提条件:
手順:
期待結果:
実装根拠:

(B) JUnit 5 + Selenium E2Eテストコード
- Javaコードを ```java ``` ブロックで提示する
- 1クラスにまとめる

(C) 仕様と実装の不整合一覧（あれば）
- 箇条書きで簡潔に列挙する
- 不足情報があればここに含めてよい

(D) 自己検証
- 次の形式で必ず出力すること:
  total_cases: <数値>
  category_counts:
    正常系: <数値>
    異常系: <数値>
    境界値: <数値>
    セキュリティ: <数値>
  duplicate_case_ids: <true/false>
  has_java_code_block: <true/false>
  junit_annotations:
    has_before_each: <true/false>
    has_after_each: <true/false>
    test_count: <数値>
  selenium_elements:
    has_webdriver: <true/false>
    has_webdriverwait: <true/false>

- 条件:
  - total_cases >= 8
  - 各カテゴリ >= 1
  - duplicate_case_ids は false
  - has_java_code_block は true
  - junit_annotations.test_count >= 3
  - selenium_elements はすべて true

- 条件を満たせない場合は、(A)(B) を修正してから (D) を出力すること。
