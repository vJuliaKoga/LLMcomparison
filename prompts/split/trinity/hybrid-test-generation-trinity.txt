以下の【業務仕様】と【UI構造コンテキスト】を入力として、対象機能のJUnit 5 + Selenium E2Eテストコードを生成してください。

【業務仕様】
{{specification}}

【UI構造コンテキスト】
{{ui_context}}

【対象機能】
{{feature}}

【目的】
- 同一入力から、JUnitで実行可能なE2Eテストコードを安定して生成する
- 仕様と実装の差分/不足情報を短く指摘する
- 無料モデルでもタイムアウトしにくいよう、出力を最小限にする

【重要方針】
- テスト設計一覧や長い説明は不要
- 最初からJUnitテストコードを生成する
- UI構造コンテキストから、対象機能に関係する要素・バリデーション・遷移・エラーメッセージだけを優先して参照する
- 不明点は推測せず「不足情報」として短く列挙する
- Javaコードは代表的な2〜3シナリオのみ実装する
- 1つのテストに複数観点を詰め込みすぎない

【指示】
1. 仕様書から対象機能の主要要件を把握する
2. UI構造コンテキストから対象機能に関係するセレクタ、入力制約、状態遷移、エラーメッセージを抽出する
3. 実行価値の高いE2Eシナリオだけを2〜3件選び、JUnitコードに直接落とし込む
4. 仕様と実装の差分・不足情報があれば短く列挙する

【必須条件（JUnitテストコード）】
- JUnit 5（org.junit.jupiter）で実行できること
- Selenium 4（WebDriver）を前提にする
- ChromeDriver を使用（起動設定はコメントで可）
- 明示待ち（WebDriverWait）を使用し、固定sleep依存は避ける
- @BeforeEach / @AfterEach を用意し、driverの初期化/終了を行う
- 代表的な2〜3件だけ実装する
- 各テストは「1テスト=1シナリオ」を基本にする
- 期待結果は assert 系で検証する
- ベースURLは System.getenv("BASE_URL") を優先し、なければ "http://localhost:8080" を使う
- Page Object は導入しなくてよい。1ファイルで簡潔にまとめる
- コードコメントは最小限にする
- セレクタは UI構造コンテキストにある id / タブ名 / 文言を優先利用する

【出力形式】
(A) JUnit 5 + Selenium E2Eテストコード
- Javaコードを ```java ``` ブロックで提示する
- 1クラスにまとめる
- 2〜3テストメソッドに絞る

(B) 仕様と実装の不整合・不足情報
- 箇条書きで3〜6件以内
- 該当なしなら「- 特になし」とだけ書く

【出力ルール】
- (A) を先に出力する
- 冗長な前置きや総括は禁止
- 自己検証セクションは出力しない
