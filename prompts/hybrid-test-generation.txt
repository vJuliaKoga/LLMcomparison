【目的】
- LLM比較のため、同一入力から「JUnitで実行可能なE2Eテストコード」を安定して生成する
- 仕様と実装の差分/不整合も検出する

【指示】
1. 仕様書からビジネス要件を理解する
2. 実装コードから実際のバリデーションルール・エラーメッセージ・状態遷移・UI要素（id/name/labelなど）を抽出する
3. 仕様と実装の差分・不整合があれば指摘する（最後に一覧化）
4. 網羅的なE2Eシナリオを作る（画面遷移・状態変化・セッション・権限変化を含める）
5. 仕様やコードから判断できない画面遷移・状態・権限は推測せず「不足情報」として列挙し、それがE2E設計に与える影響を説明する

【必須条件（テスト設計）】
・最低15件以上生成
・正常系 / 異常系 / 境界値 / セキュリティ を必ず含める
・各カテゴリ最低3件以上
・重複禁止
・金融システム特有のリスク（不正試行、金額境界、権限、二重送信、ロック、状態遷移）を考慮

【必須条件（JUnitテストコード）】
- JUnit 5（org.junit.jupiter）で実行できること
- E2E UIテストとして Selenium 4（WebDriver）を前提にコードを生成する
  - ChromeDriver を使用（起動設定はコメントで可）
  - 明示待ち（WebDriverWait）を使用し、固定sleep依存は避ける
- @BeforeEach / @AfterEach を用意し、driverの初期化/終了を行う
- 各テストは「1テスト=1シナリオ」を基本に、読みやすい命名にする
- 期待結果は assert系（assertEquals / assertTrue 等）で検証する
- 仕様・実装に根拠がある検証には、コメントで「仕様の該当箇所」または「コードの該当箇所（引用）」を添える
- ベースURLは環境依存なので、次のいずれかで扱えるようにする
  - System.getenv("BASE_URL") があればそれを使い、なければ "http://localhost:8080" をデフォルトにする

実装根拠: (コードの該当箇所を引用)

(B) JUnit 5 + Selenium E2Eテストコード
- Javaコードを ```java ``` ブロックで提示（コンパイル可能な形）
- 可能なら Page Object を簡易導入（任意）

(C) 仕様と実装の不整合一覧（あれば）
- 箇条書きで列挙し、E2E設計への影響も一言添える

(D) 自己検証（Validation Checklist）
- 次の形式で必ず出力すること（機械検証用）:
  total_cases: <数値>
  category_counts:
    正常系: <数値>
    異常系: <数値>
    境界値: <数値>
    セキュリティ: <数値>
  duplicate_case_ids: <true/false>
  has_java_code_block: <true/false>
  junit_annotations:
    has_before_each: <true/false>
    has_after_each: <true/false>
    test_count: <数値>
  selenium_elements:
    has_webdriver: <true/false>
    has_webdriverwait: <true/false>

- 条件:
  - total_cases >= 15
  - 各カテゴリ >= 3
  - duplicate_case_ids は false
  - has_java_code_block は true
  - junit_annotations.test_count >= 3
  - selenium_elements はすべて true

- もし条件を満たせない場合は、(A)(B) を修正してから (D) を出力すること。
