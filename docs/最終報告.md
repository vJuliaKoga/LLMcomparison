# 無料LLM性能比較レポート

## OpenRouter + Promptfoo による JUnit 5 + Selenium E2E テストコード生成検証

## 前提

- OpenRouter無料枠だとトークン制限がきつい為、下記対応を行った。
  - 3機能に対して3つのテストケース生成×10回で検証
  - 仕様・プロンプトも分割
- API接続が不安定で全件出力完了に至っていないモデルも含まれている。

---

## 概要

対象タスク：「業務仕様・UI構造コンテキストを与えた上で、JUnit 5 + Selenium ベースの E2E テストコードを生成する」こと。
対象機能： ログイン / 振込 / 監査ログ の 3 機能。
評価は複数のパイプライン段階（出力構造検証 → 仕様カバレッジ検証 → 静的構文検証 → コンパイル → DeepEval → アクション抽出 → Playwright 実行）を通じて実施。

評価軸は下記6項目。

- Javaコードブロックの存在

- `@Test` 2本以上
- `Thread.sleep` 不使用
- ダミー assertion 不使用

### Promptfoo で明らかになった主な失敗パターン

**`WebDriverWait` 欠落（cohere）**
ログイン・振込の全 20件が `WebDriverWait` 不使用として fail。
暗黙待ち（`implicitlyWait`）を使用する独自実装に傾く傾向が、ここでも数値として確認された。

**`auditor` ロール欠落（llama / trinity）**
監査ログ機能で「admin / auditor / customer の 3ロールをカバーしているか」のアサーションが全件 fail。
仕様書から `auditor` の存在は読み取れているが、生成コードのテストシナリオ選定で省略される傾向がある。

**Javaコードブロック・`@Test` の欠落（glm）**
ログイン・監査ログで Javaコードブロック自体が出力されないケースが多発。DeepEval での全件 fail と一致しており、出力の根本的な形式不整合が確認される。

**gemma のキャッシュ応答**
平均レイテンシが 0.03s と極端に低く、実測値ではなくキャッシュ応答の可能性が高い。Promptfoo 上の pass 率は高いが、実推論時の再現性は別途確認が必要。

## ② 曖昧なプロンプトが与えられた場合、何を重要と判断する傾向

本検証では「手動ケース流用を前提に、指示がやや曖昧でもどう解釈するか」という趣旨のプロンプトを使用した。以下にモデルごとの解釈傾向を示す。

### 指示形式・テンプレート構造の遵守を優先するモデル

**該当：`llama-3.3-70b-instruct` / `trinity-large-preview` / `qwen3-next-80b-a3b-instruct`**

プロンプトの出力形式（`(A)` Javaコード、`(B)` 不足情報の列挙）を優先的に守る傾向。
仕様書の構造を読み取り、求められたセクション構成に沿って出力を組み立てる。
曖昧な部分については「不足情報」として列挙する指示に素直に従い、推測での補完を控える。
特に `qwen` と `llama` は仕様カバレッジが全件通過しており、「何をテストすべきか」の読み取り精度が高い。

### 実装方針を独自解釈で補完するモデル

**該当：`command-r7b-12-2024`（Cohere）**

コードは生成できるが、待機戦略（`WebDriverWait` の使い方）やロケータ設計に独自解釈が入る。
曖昧な指示に対して「推測で補完する」方向に動きやすく、仕様との乖離が生じやすい。
`locator` の解決 Warning が 90件と突出しており、「自分なりに書いた結果セレクタが合わない」ケースが多い。

### 実行可能性・コンパクトさを優先するモデル

**該当：`mistral-small-3.1-24b-instruct` / `gpt-oss-20b`**

仕様の観点を広く拾うより、「実際に動くコードを書く」方向に寄る。
コンパイル成功率が高く、Playwright 実行でも安定（特に Mistral）。
曖昧な仕様への対処は「観点を絞って確実に動くシナリオを書く」方向で、カバレッジよりも実行確実性を選ぶ。

### 思考が長大化しトークンを消費しすぎるモデル

**該当：`glm-4.5-air` / `gemma-3n-e4b-it`**

曖昧な指示に対して「考えすぎる」方向に動く。
思考のトークン消費が大きく、出力が途中で切断される。
トークン上限を増やしても、その分さらに思考に使うため根本解決にならない。

## 評価横断まとめ：モデルの位置づけ

```
実行可能性（Playwright pass）
高　│ Mistral ◎
    │
    │
    │
    │
低　│ Qwen, Llama, Trinity, GPT-oss, Cohere, Gemma, GLM
    └──────────────────────────────────────────────────
      低　　　仕様カバレッジ（validate-spec）　　高
              ↑Mistral 低　　↑Qwen / Llama / Trinity 高
```

- **仕様を読む × 実行まで通す** の両立はいずれのモデルも達成できていない
- `Mistral` は「実行まで通す」側に特化、`Qwen / Llama / Trinity` は「仕様を読む」側に特化
- 実用には「仕様カバレッジの高いモデルでドラフト生成 → Mistral 的な絞り込みで実行コードに整形」といった 2段構えが有効な可能性がある

## 補足：API 可用性と遅延設定の影響

| モデル          | 無料経路の安定性                     | 備考                             |
| --------------- | ------------------------------------ | -------------------------------- |
| Llama 系        | delay 15000ms で安定化               | 遅延を増やすと応答取得が安定     |
| Mistral 系      | delay 15000ms で一時安定するが不安定 | 無料経路が揺れやすい             |
| Groq 経由       | 応答は通りやすいが制限強め           | レート制限に注意                 |
| Gemini / GLM 系 | 思考トークン消費が大きく途中切断     | トークン増加は根本解決にならない |

> **注意：** 今回の結果は「モデルの能力差」と「無料 API 経路の可用性差」が混在している。特に Llama / Mistral については、遅延設定や可用性の影響が大きく、純粋なモデル性能の比較とは切り離して見る必要がある。

## まとめ

今回の検証で最も示唆的だったのは、**「テストコードとして書かれていること」と「実ブラウザ上で最後まで通ること」は全く別の問題**だという点。

仕様カバレッジや DeepEval で高評価を得た `qwen` や `llama` は、Playwright 実行段階では pass ゼロ。
逆に仕様カバレッジが低い `mistral` が唯一の Playwright 全件 pass を達成した。

また、Promptfoo の追加データにより、**評価軸によって「優秀なモデル」が入れ替わる**ことが明確になった。
Promptfoo アサーション（形式要件の満足）では `qwen` / `groq` / `gemma` が満点を記録する一方、
DeepEval（代表シナリオの妥当性）では `qwen` が最高位、`gemma` は下位。
`llama` と `trinity` は Promptfoo で監査ログの `auditor` ロールを一貫して省略しており、「仕様は読めているが選定で落としている」傾向が数値として確認された。

無料モデルを E2E テストコード生成に使う場合、以下の使い分けが現実的と考えられる。

- **テスト設計・観点洗い出し** → `qwen3-next-80b-a3b-instruct` / `llama-3.3-70b-instruct`
- **形式要件を確実に満たすコード生成** → `groq`（ただしレート制限に注意）
- **実行可能コードの生成** → `mistral-small-3.1-24b-instruct`（サンプル数は少ないが最も安定）
- **コンパイル品質が必要な骨格生成** → `gpt-oss-20b`
- **現状では実用困難** → `gemma-3n-e4b-it`（Promptfoo は pass だがキャッシュ疑惑あり、他指標はほぼ全滅）/ `glm-4.5-air`
