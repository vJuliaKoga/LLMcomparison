## 2. `validation/validate-output.mjs`

### 何をしたか

- `output.txt` の構造を検証した
- 現行プロンプトに合わせて、判定基準を調整した
  - `(D) 自己検証` は通常は要求しない
  - `mode=current` では `(B) 仕様と実装の不整合・不足情報` を必須とする
  - `mode=current` の `@Test` 本数は **2本以上**

- 吸収できる形式は以下
  - legacy: `(A)` テストケース列挙あり
  - current strict: Javaコード + `(B)` 見出しあり
  - current loose: Javaコード + `**仕様と実装の不整合・不足情報**` のような緩い見出し

- `カテゴリ` / `優先度` の表記ゆれも吸収するよう調整した
- `--json` 出力に対応させ、全件集計できるようにした

### 分かったこと

- 初期状態では `ajv` / `ajv-formats` 依存が不足しており実行できなかった
- 依存導入後は動作したが、最初の parser は実データを十分に吸収できていなかった
  - 例: `カテゴリ: 正常系` を `[正常系]` 前提でしか読めなかった

- 修正後の `validate-output-summary.json` では、集計結果は以下の通りだった
  - 対象ファイル数: **220件**
  - Valid: **110件**
  - Error: **110件**
  - `mode=current`: **110件**
  - `mode=legacy`: **0件**
  - Warning: **0件**

- 主なエラー内訳は以下
  - `(B)` セクション不足による schema エラー: **50件**
  - `Missing code fence.`: **40件**
  - `Missing required Java snippets: WebDriverWait`: **20件**

### 解釈

- `validate-output` は「形式チェック」としては機能しているが、**全件通過には至っていない**
- 特に current 形式では
  - `(B)` セクション欠落
  - Javaコードブロック欠落
  - `WebDriverWait` 欠落
    が主な失敗要因になっている

- 旧版で大量に出ていた `(D) 自己検証` 欠落 warning は、現行プロンプト基準では除外できた
- したがって、現在の `validate-output` は
  - **現行プロンプトに合わせた形式チェック**
  - **Java構造の最低条件確認**
    の役割を持つ

## 3. `validation/schema.json`

### 何をしたか

- `validate-output.mjs` が組み立てた解析結果に対する schema を定義した
- `mode=legacy / current` ごとの最低条件を持たせた
  - legacy:
    - `(A)` あり
    - テストケースあり
    - `@Test >= 3`

  - current:
    - `(B)` あり
    - Javaコードブロックあり
    - `@Test >= 2`

### 分かったこと

- schema 自体は「最終チェック」として機能している
- 初期段階では parser が実データに追いついておらず、schema failure が多発した
- 現在は parser の吸収力を上げたことで、**schema はより現行プロンプトに沿った失敗理由を返す段階**になった
- ただし current 出力の中には `(B)` 欠落や code fence 欠落が残っており、schema failure は依然発生している

### 解釈

- schema は「形式評価の境界線」を定義する役割を持つ
- 問題の本質は当初は parser 側にあったが、現時点では
  - プロンプト出力自体の揺れ
  - モデルごとの形式遵守の差
    を可視化する役割に移っている
